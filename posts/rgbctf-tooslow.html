rgbCTF - Too Slow

Resolution of the rgbCTF challenge named Too Slow

July 13, 2020 - 4 minute read -
Ctf Rev rgbCTF

Too Slow is an easy reverse engineering challenge from rgbCTF, it was a funny challenge because I’m pretty new to reverse engineering techniques and it helped me a lot, I played this CTF just for fun with my team, 0x8Layer :)

The challenge had the following description/details:

The following executable file is available on: a.out

Ok, now we need to check the executable in order to determinate his characteristics:

This is the output after executing the binary:

After that we need to take a deep look inside the binary, to do it I’m going to use the Ghidra tool:

With Ghidra debugger, we noticed three interesting functions:

main

getKey

win

The functions’s graph:

Explanation

The uVar1 variable takes the result of getKey() function and send it to win function. There’s a problem, the getKey function would take a long time to pass through the while loop because it increments by 1 the local_10 variable until the number 0x265d1d23 or 643636515 in decimal.

Exploiting

We can jump straight to the win function by calling this function directly through gdb, to do it we need to open the vulnerable executable on gdb:
